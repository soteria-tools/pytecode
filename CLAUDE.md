# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

pytecode turns Python source into pure OCaml data structures by reusing CPython's own parser and compiler (pinned to **3.13**). It never reimplements compilation. It is the foundation for a Python interpreter written in OCaml.

## Commands

The project has a **local opam switch** in `_opam/`. If switch hooks aren't active, prefix dune with `opam exec --switch . --`.

```sh
dune build                  # never needs Python
dune fmt                    # the project uses ocamlformat (version pinned in .ocamlformat);
                            # run before committing
dune runtest                # golden expect-tests; needs python3.13 on PATH
dune promote                # accept changed golden output (review the diff first!)
dune build @stdlib          # acceptance gate: dumps + decodes + Phir-transforms the
                            # entire 3.13 stdlib (~10 s); run after any change to
                            # dump_bytecode.py, Decode, or Phir
dune build @gen             # checks generated lib/opcode.ml(i) for drift vs the pinned
                            # CPython (runs python3.13 + ocamlformat); fix with dune promote
dune exec pytecode -- dump foo.py    # dis-like pretty-print of the bytecode AST
dune exec pytecode -- phir foo.py    # pretty-print the Phir IR
dune exec pytecode -- json foo.py    # raw JSON envelope (debugging)
```

Run a single test file or test (from `_build/default/test`, after `dune build`):

```sh
cd _build/default/test && ./.pytecode_tests.inline-tests/inline-test-runner.exe \
  inline-test-runner pytecode_tests -only-test 'test_phir.ml:87'   # or just test_phir.ml
```

Python discovery: `$PYTECODE_PYTHON`, else `python3.13` on PATH. The stdlib sweep and golden tests are pinned to CPython 3.13 output; goldens can shift between 3.13.x patch releases (rare).

## Architecture

```
source.py → tools/dump_bytecode.py (pinned 3.13) → versioned JSON envelope
              │ run by lib/subprocess.ml (default backend; batch NDJSON mode too)
              ▼
        lib/decode.ml (yojson) → Ast.code  ──(lib/phir.ml)──►  Phir.code
```

- **`Ast.code`** mirrors CPython code objects with exactly **one normalization** (specified in `doc/normalization.md`, the contract any future backend must honor): `CACHE`/`EXTENDED_ARG` stripped, args EXTENDED_ARG-folded, and all jump args + exception-table boundaries are **absolute instruction indices**, not byte offsets. Everything else is raw CPython semantics (args index `consts`/`names`/`localsplus`; flag bits like LOAD_GLOBAL's push-NULL bit preserved).
- **Backends** implement `Backend_intf.S` and all return `Ast.code`, so every backend is validated by the same tests. `identity ()` must capture everything affecting output (python version, script hash, config, wire format) — `Cache.wrap` keys on it (BLAKE-256 of identity+path+source; Marshal values; transparent re-dump on any mismatch). A future pyml backend should run `Dump_script.source` in-process and reuse `Decode` verbatim; a native `.pyc` reader would construct `Ast.code` directly per the normalization contract.
- **`Phir`** (Python High IR) is derived from `Ast.code`: statically known operands (constants, variable reads, NULL) fold into consuming instructions; `NOP`/`RESUME` dropped; superinstructions expanded; operator/intrinsic/flag args decoded into variants. Soundness rules live in `lib/phir.mli`: folding only captures a contiguous run of pure reads immediately before the consumer, never across a jump target or exception-table boundary; `Stack` operands always form a prefix of an operand list (push order; pop right-to-left); folded operands evaluate left-to-right at the instruction. Operand orders follow CPython's push order — check `Lib/dis` docs / bytecodes.c before changing any.

## Generated files — do not hand-edit

- `lib/opcode.ml(i)`: generated by `tools/gen_opcodes.py` from the pinned CPython, **checked in**. It is just the variant with flag attributes (`| LOAD_CONST [@has_arg] [@has_const]`) + `[@@deriving opcode, eq, ord]`; the in-tree ppx (`ppx/ppx_opcode.ml`, deriver name `opcode`) generates `to_string`/`of_string`/`all`/`has_*` from the attributes. Regenerate: `dune build @gen && dune promote`. The `@gen` rule pipes generator output through ocamlformat before diffing, so the checked-in files stay formatted like every other source.
- `lib/dump_script.ml`: `tools/dump_bytecode.py` embedded as a string, regenerated on every build (a dune `cat` rule) — can never go stale.

Opcode **names** (not numbers) are the wire format, so CPython renumbering between versions is irrelevant.

## Conventions and invariants to preserve

- The opcode match in `Phir.of_code` is **exhaustive with no wildcard** — intentionally, so a CPython version bump fails at compile time. Keep it that way.
- `tools/dump_bytecode.py` uses only `dis` APIs that are public/stable since 3.13, plus a vendored exception-table varint parser. Any wire-format change must bump `FORMAT` there **and** `Decode.format_version`, and will invalidate caches (by design). Its `--check` flag cross-validates against CPython private helpers — use it when touching extraction logic.
- New test modules must be added to the explicit `(modules ...)` list in `test/dune` (the sweep executable is kept out of the inline-tests library).
- Golden tests are the executable spec; when promoting, eyeball against `python3.13 -m dis` (for `dump`) or the Phir contract. The stdlib sweep asserting zero failures / zero unknown opcodes is the real acceptance gate.

## Bumping the pinned CPython (e.g. 3.14)

1. Update `EXPECTED` in `tools/dump_bytecode.py` and `tools/gen_opcodes.py`, and `Decode.expected_python_prefix`.
2. `dune build @gen && dune promote` under the new interpreter (the variant becomes a union over supported versions).
3. The compiler will flag new opcodes in `Phir.of_code`'s match; handle them.
4. `dune runtest` (review/promote shifted goldens) and `dune build @stdlib` must be green.
